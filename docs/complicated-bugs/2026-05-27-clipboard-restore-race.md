# 剪贴板上屏粘贴旧内容

日期：2026-05-27  
相关范围：剪贴板上屏、Windows Ctrl+V、目标应用事件时序

## 现象

VoxType 识别到新文本后请求上屏，但目标输入框中粘贴出来的是旧剪贴板内容，而不是刚识别出的文本。

## 为什么复杂

从 VoxType 角度看，上屏步骤已经执行：写剪贴板、发送 `Ctrl+V`、恢复剪贴板。问题出在目标应用处理 `Ctrl+V` 的时间可能晚于 VoxType 恢复剪贴板的时间。

这是跨进程时序问题，不是单纯的函数调用顺序问题。

## 根因

旧策略类似：

1. 读取旧剪贴板。
2. 写入识别文本。
3. 发送 `Ctrl+V`。
4. 立即恢复旧剪贴板。

Windows 目标应用可能在第 4 步之后才真正读取剪贴板。因此它读到的是恢复后的旧内容。

## 修复

MVP 阶段选择可靠性优先：

- 写入剪贴板后读回校验。
- 发送 `Ctrl+V` 后不立即恢复旧剪贴板。
- 让系统剪贴板保留本次识别文本，优先保证目标输入框读到正确内容。

核心代码位置：

- `src-tauri/src/insertion.rs`

## 后续经验

- 跨进程输入不能假设 `Ctrl+V` 是同步完成的。
- 剪贴板恢复要么延迟、要么等待目标应用确认，但 MVP 中没有可靠跨应用确认机制。
- 后续更稳路线是 `SendInput(KEYEVENTF_UNICODE)`、TSF 或 IME 通道，避免剪贴板作为主上屏通道。
